# Development Setup

## Mock mode vs. live mode

ComplianceWatch ships in **mock mode** by default — no API key required.

| Mode | How to run | What it does |
|---|---|---|
| **Mock (demo)** | Open `frontend/demo.html` in any browser | Pre-written audit reports with simulated streaming, policy panels, risk badges, and citations. Zero API calls. |
| **Live (API)** | `python frontend/app.py` + API key in `.env` | Real Claude analysis via SSE streaming. Report text is generated by Claude; requires `ANTHROPIC_API_KEY`. |
| **Full pipeline** | `python -c "from pipeline import auditar_documento; ..."` | Runs all 4 stages: routing → entity extraction → parallel policy checks → draft+citation review. Requires both API keys. |

### What mock mode shows vs. what live mode produces

The mock demonstrates the **full pipeline output** — the report format, risk badge, policy panel (FLAG/PASS/N/A), and citations panel that the complete pipeline (`auditar_documento`) would produce. The live SSE mode (`Run Audit` button) streams a direct Claude response, which produces an equivalent audit report but without the structured side panels — those require the full pipeline.

## Going from mock to production-ready

### Step 1 — API keys

```bash
cp .env.example .env
```

Edit `.env`:
```
ANTHROPIC_API_KEY=sk-ant-...
VOYAGE_API_KEY=pa-...        # only needed for RAG / full pipeline
```

### Step 2 — Verify the setup

```bash
uv sync
python scripts/smoke_test.py
```

Expected:
```
[PASS] basic call: 'OK'
[PASS] multi-turn: '...42...'
[PASS] streaming: received N chunks
All tests passed.
```

### Step 3 — Run the web UI

```bash
python frontend/app.py
```

Open `http://localhost:8080`. Paste any invoice or contract text, click **Run Audit**. The response streams token by token from Claude.

### Step 4 — Run the full pipeline (optional)

To exercise all 4 pipeline stages (routing, entity extraction, parallel checks, report chain):

```bash
python -c "
from pipeline import auditar_documento
import json, textwrap

result = auditar_documento(open('tests/fixtures/invoice_cortex_005.txt').read())
print('Doc type:', result['doc_type'])
print('Policy results:')
for p in result['policy_results']:
    print(' ', p)
print()
print(textwrap.fill(result['report'], 80))
"
```

### Step 5 — Run the deep-investigation agent

Triggered manually on already-flagged cases:

```bash
python agent/investigacao_profunda.py \
  --case "Duplicate invoice suspected from Cortex Supply Chain" \
  --document tests/fixtures/invoice_cortex_005.pdf
```

The agent will call compliance tools, ask for analyst input if needed, and use extended thinking automatically when its confidence drops below threshold.

### Step 6 — Run the evaluation pipeline

```bash
python evaluation/generate_dataset.py           # generates evaluation/dataset.json
python evaluation/run_eval.py \
  --dataset evaluation/dataset.json \
  --report evaluation/reports/baseline.json
```

The evaluation pipeline grades each case with a model grader (prefill-enforced output order) and a code grader (citations hard gate). Results are averaged and checked against the DoD threshold (≥ 7.0).

### Step 7 — MCP server

```bash
mcp dev mcp_server/server.py       # inspector UI
claude mcp add compliance uv run mcp_server/server.py   # register with Claude Code
```

## Merge strategy

**All merges use the default merge commit** (not squash). This preserves per-phase commit history, which is part of the portfolio value. Set this once and never change:

```bash
git config merge.ff false
```

Do not use squash merge — the atomic commits within each phase are meaningful.

## Branch lifecycle

1. Branch from `main`: `git checkout -b phase-N-slug`
2. Implement atomically (one commit per component)
3. Push and open PR referencing the issue (`Closes #N`)
4. Merge → delete branch
5. Next phase begins only after the previous PR is merged
